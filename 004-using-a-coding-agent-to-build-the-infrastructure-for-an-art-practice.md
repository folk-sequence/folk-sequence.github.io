This entire channel runs on a small program I built with a coding assistant called Claude Code. The program is a command-line tool, which is the kind of program you run by typing its name into a terminal window, and its name is folkseq. What folkseq does is take the screen recording I make of each Bitwig session, convert the recording into the format YouTube expects, generate a thumbnail through an image generation service, upload the video, schedule it to publish at three in the afternoon Central time on the next available day, and update the queue so I can see what is scheduled for when. I record the song. The tool handles everything that comes after.

I did not write the tool in the usual sense. I described what I wanted to a coding assistant in plain English, and the assistant wrote the code, branch by branch, while I reviewed what it produced and asked for changes when I did not like the result. Total time from the moment I decided I needed an automated pipeline to the moment the pipeline was running in production: about a day. I had no existing scripts. I had not built anything like this before. I had a Mac, an external drive full of recordings, and a YouTube account.

This is not the use case anyone was selling Claude Code for. The marketing for this category of tool is aimed at software engineers shipping features at companies with code review processes and test suites and continuous integration pipelines. I am a musician who wanted his upload workflow to disappear so he could focus on making songs. The assistant did not care about the difference. Code is code. The assistant built me an upload pipeline with the same patient competence it would have used to build a feature for a software company, and the result is the same.

The thing I want to write about is not the specific tool, which is small and unremarkable. The thing I want to write about is the larger pattern, which is that a coding assistant collapses the time between deciding you need infrastructure for your art practice and actually having the infrastructure, and that this collapse is more important for non-engineers than for engineers, which is the opposite of the way the tools are being marketed.

Here is the underlying principle. Most artists I know burn an enormous amount of energy on the surrounding tasks of their practice. The surrounding tasks are things like file management, format conversion, metadata, scheduling, social media posting, backup, archival, distribution. None of these tasks are the art. All of them have to happen for the art to reach anyone. The traditional options have been to do them manually, which eats hours every week, or to pay a service to do them, which costs money and usually does the wrong thing, or to learn to write code, which takes months and is what most artists were specifically trying not to do when they decided to be artists.

A coding assistant is a fourth option that did not exist a year ago. You describe the surrounding task in plain English, the assistant produces a tool that does the surrounding task, and then the surrounding task is gone. Not reduced. Gone. You no longer think about it. The mental space it occupied gets returned to you, and you spend that mental space on the art.

The opposite of this situation is the artist with no infrastructure, who spends Sunday afternoon converting files and writing descriptions and clicking through upload forms and forgetting to schedule things and missing days because the friction is too high to bother. The similar situation, which is not the same situation, is the artist who pays a manager or a virtual assistant to do those things. The manager option is real and sometimes the right choice, but it has its own costs. The manager has to be paid. The manager has to be told what to do. The manager makes decisions that drift from what the artist would have decided. The manager creates a layer of distance between the artist and the operational reality of the practice, and that distance sometimes becomes an excuse for the artist to not understand their own practice.

A coding assistant is closer to the manager model than the do-it-yourself model in terms of how much work it removes from the artist. It is closer to the do-it-yourself model in terms of how much control the artist retains. You read the code, or you skim it, or you have the assistant explain what it did. You can change things. You can run the tool yourself. You know exactly what is happening on your behalf because you authorized every part of it. The tool is yours in a way that a service is not yours.

There is a specific objection that comes up when I describe this to people, which is that letting an assistant write code that runs on your behalf is risky because you do not know what is in it. The objection is correct as a description of risk and wrong as a reason not to do it. The risk is real. You should read the code. You should not let the assistant do anything destructive without checking. You should treat the assistant the way you would treat a junior engineer you just met, which is to say with patience and oversight and reasonable trust. The risk is not different in kind from the risk of running any other software you did not write yourself. It is different in degree only because you understand it less, and the way to fix that is to look at the code.

The specific things I would tell another artist trying this. The assistant is patient. You can iterate on a description until you get what you want. The assistant does not get tired or annoyed or sigh at you. Be specific about the things that matter to you. The phrase one video per day at three in the afternoon Central time is a real specification. The phrase schedule them daily is a wish. Read the code before you trust it, not all of it, but enough to know what is being run on your behalf and what would happen if it ran the wrong way. Do not be precious about throwing things out. It is faster to delete a half-working version and regenerate than to argue with the assistant about why the half-working version is wrong.

The tool that runs this channel is not sophisticated. It is just gone, in the sense that I never have to think about it. That is the entire goal of infrastructure. The assistant let me get there in a day, starting from nothing, with no code-writing skill of my own. That is the part that is new.
